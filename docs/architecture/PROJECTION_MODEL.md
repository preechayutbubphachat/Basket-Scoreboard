# PROJECTION_MODEL.md

## 0. Document Status

**Document type:** Architecture / Read Model / Projection Specification  
**System:** Basketball Scoreboard and Tournament Management Web Application  
**Default rule profile:** FIBA  
**Database target:** MariaDB with InnoDB  
**Hosting target:** Hostatom / Plesk Shared Hosting compatible mode  
**Realtime strategy:** Polling-first, Socket.IO optional  
**Source of truth:** `match_events`

---

## 1. Purpose

This document defines how the system builds fast, UI-friendly read models from append-only match events.

The UI must not calculate official match state from scratch on every screen render. Instead, backend services must maintain rebuildable projections derived from `match_events`.

The purpose of projections is to support:

- live public scoreboard
- operator score control screen
- foul control screen
- timeout control screen
- clock and shot clock display
- match summary
- replay timeline
- tournament standings
- reconnect and polling-based realtime synchronization
- post-game review
- audit-safe correction display

[SYSTEM RECOMMENDATION] Projections are optimized views. They are not the source of truth.

---

## 2. Core Principles

## 2.1 Match Events Are the Source of Truth

[SYSTEM RECOMMENDATION] The only authoritative match history is the ordered event stream in `match_events`.

```txt
match_events
  -> projection builder
  -> live_scoreboard_projection
  -> operator_score_projection
  -> foul_projection
  -> timeout_projection
  -> clock_projection
  -> shot_clock_projection
  -> match_summary_projection
  -> replay_timeline_projection
  -> tournament_standings_projection
```

## 2.2 Projections Are Rebuildable

Every projection must be rebuildable from `match_events`.

If a projection is corrupted, stale, or generated by buggy code, the system must be able to delete and rebuild it without losing official match history.

## 2.3 Projection Bugs Must Not Corrupt Event Store

Projection code must never update, delete, reorder, or reinterpret historical events.

Allowed:

- read events
- apply events to read model
- update projection rows
- update projection checkpoint
- mark projection as stale
- rebuild projection from event stream

Not allowed:

- mutate old event payload
- delete event rows
- modify `seq_no`
- hide correction events
- create official state without appending event
- use projection state as source of truth for audit

## 2.4 Snapshots Are Optimization Only

`match_snapshots` can speed up rehydration, but snapshots are not authoritative.

If snapshot and event stream disagree, rebuild from event stream.

## 2.5 Projection Reads Must Include Sequence

Every projection response must include:

```ts
{
  matchId: string;
  lastEventSeq: number;
  projectionVersion: number;
  generatedAt: string;
}
```

This allows clients to know whether their UI is up to date.

## 3. Definitions

## 3.1 Event Store

The append-only table containing official match events.

Example table:

```txt
match_events
```

## 3.2 Projection

A database read model built from match events for a specific UI or query purpose.

Examples:

```txt
live_scoreboard_projection
foul_projection
match_summary_projection
```

## 3.3 Snapshot

A compact state representation at a specific event sequence.

Used for fast rehydration.

Example:

```txt
match_snapshots.last_seq_no = 120
match_snapshots.state_json = {...}
```

## 3.4 Checkpoint

A marker showing the latest event sequence applied to a projection.

Example:

```txt
projection_checkpoints.projection_name = live_scoreboard_projection
projection_checkpoints.match_id = match_123
projection_checkpoints.last_applied_seq = 120
```

## 3.5 Replay

A timeline view generated from match events, including corrections.

Replay must preserve correction history.

## 3.6 Polling Cursor

The client-side `lastEventSeq` used to request changes after the latest known event.

Example:

```http
GET /api/matches/:matchId/events?afterSeq=120
```

## 4. Projection Design Policy

## 4.1 UI Must Read Projections, Not Raw Events

Default UI reads:

```txt
GET /api/matches/:matchId/live-state
GET /api/matches/:matchId/operator-state
GET /api/matches/:matchId/fouls
GET /api/matches/:matchId/summary
GET /api/tournaments/:tournamentId/standings
```

Raw event reads are allowed for:

- replay screen
- audit screen
- debugging
- rebuild job
- correction review

## 4.2 Projection Update Timing

For Hostatom / Plesk / MariaDB compatible deployment, projections should be updated synchronously inside the command transaction where possible.

Preferred transaction flow:

```txt
1. Receive command
2. Authenticate user
3. Authorize user
4. Validate command payload
5. Load current match stream version
6. Validate expectedSeq
7. Run rules engine
8. Append match event
9. Apply event to affected projections
10. Update projection checkpoint
11. Commit transaction
12. Return latest projection state or event result
```

## 4.3 Background Workers Are Optional

Because shared hosting may not support reliable long-running background processes, projection updates must not depend on permanent workers.

Allowed modes:

```txt
Mode A: Synchronous projection update in request transaction
Mode B: Lazy rebuild when projection is stale
Mode C: Manual admin rebuild endpoint
Mode D: Background worker only when deployed on VPS/cloud/local server
```

[SYSTEM RECOMMENDATION] In Hostatom shared hosting mode, use Mode A as the default.

## 4.4 Projection Consistency Level

Live match control requires strong consistency for operator screens.

Viewer/public screens can tolerate small delays.

Recommended:

```txt
Operator projection: synchronous update
Public scoreboard projection: synchronous update or immediate pull after event
Tournament standings projection: update when match becomes official, not every score event
Replay projection: rebuild/lazy generate from events
```

## 5. Main Projections

## 5.1 `live_scoreboard_projection`

### Purpose

Read model for the public scoreboard display.

### Used By

- public scoreboard screen
- spectator display
- livestream overlay
- match venue display
- remote viewer page

### Must Include

```ts
type LiveScoreboardProjection = {
  matchId: string;
  tournamentId?: string;
  courtId?: string;

  homeTeam: {
    teamId: string;
    displayName: string;
    shortName?: string;
    logoUrl?: string;
    score: number;
    teamFouls: number;
    timeoutsRemaining: number;
  };

  awayTeam: {
    teamId: string;
    displayName: string;
    shortName?: string;
    logoUrl?: string;
    score: number;
    teamFouls: number;
    timeoutsRemaining: number;
  };

  period: {
    type: "REGULATION" | "OVERTIME";
    number: number;
    label: string;
    isHalftime: boolean;
    isInterval: boolean;
  };

  gameClock: {
    status: "NOT_STARTED" | "RUNNING" | "STOPPED" | "EXPIRED";
    remainingMs: number;
    startedAtServerTime?: string;
    serverNow: string;
  };

  shotClock: {
    status: "VISIBLE" | "RUNNING" | "STOPPED" | "OFF" | "EXPIRED";
    remainingMs?: number;
    startedAtServerTime?: string;
    serverNow: string;
  };

  possession: {
    teamId?: string;
    arrowTeamId?: string;
    direction?: "LEFT_TO_RIGHT" | "RIGHT_TO_LEFT";
  };

  matchStatus:
    | "SCHEDULED"
    | "WARMUP"
    | "IN_PROGRESS"
    | "PERIOD_BREAK"
    | "HALFTIME"
    | "OVERTIME_PENDING"
    | "FINISHED"
    | "OFFICIAL"
    | "CORRECTED";

  lastEventSeq: number;
  projectionVersion: number;
  generatedAt: string;
};
```

### Event Inputs

Must react to:

```txt
MATCH_CREATED
MATCH_STARTED
PERIOD_STARTED
PERIOD_ENDED
OVERTIME_STARTED
SCORE_ADDED
SCORE_REMOVED_BY_CORRECTION
SCORE_CORRECTED
PLAYER_FOUL_ADDED
TEAM_FOUL_ADDED
TIMEOUT_GRANTED
TIMEOUT_CANCELLED_BY_CORRECTION
GAME_CLOCK_STARTED
GAME_CLOCK_STOPPED
GAME_CLOCK_SET
SHOT_CLOCK_STARTED
SHOT_CLOCK_STOPPED
SHOT_CLOCK_RESET_24
SHOT_CLOCK_RESET_14
SHOT_CLOCK_SET
SHOT_CLOCK_OFF
POSSESSION_CHANGED
DIRECTION_SWITCHED
MATCH_FINISHED
MATCH_MARKED_OFFICIAL
CORRECTION_APPLIED
```

### Projection Rules

- Score must never be negative.
- Displayed score must reflect correction events.
- Team fouls must show current period count.
- Timeout remaining must be computed from rule profile and granted timeouts.
- Clock must use deadline-based calculation.
- Public clients must not receive private audit details.
- Must include `lastEventSeq`.

## 6. Projection Tables for MariaDB

## 6.2 JSON Column Policy

JSON is acceptable for projection state because projections are rebuildable and optimized for read shape.

Do not use JSON-only storage for source-of-truth event metadata that must be indexed.

Recommended source-of-truth event columns:

```txt
event_id
match_id
seq_no
event_type
actor_user_id
actor_role
device_id
occurred_at
recorded_at
correlation_id
causation_id
expected_seq
rule_profile_id
payload_json
```

## 7. Projection Builder Contract

## 7.1 Pure Apply Function

Projection builders should use pure functions where possible.

```ts
type ProjectionApplyContext = {
  ruleProfile: RuleProfile;
  previousState: unknown;
  event: MatchEvent;
};

type ProjectionApplyResult = {
  nextState: unknown;
  affectedProjectionNames: string[];
  warnings: ProjectionWarning[];
};
```

## 7.2 Projection Warning

```ts
type ProjectionWarning = {
  code:
    | "UNKNOWN_EVENT_TYPE"
    | "MISSING_RULE_SOURCE"
    | "STALE_PROJECTION"
    | "INCONSISTENT_EVENT_PAYLOAD"
    | "REBUILD_REQUIRED";
  message: string;
  eventSeqNo?: number;
};
```

## 7.3 Projection Builder Must Not

Projection builder must not:

- call external APIs
- append match events
- change permissions
- mutate historical event payloads
- hide audit data from admin projection
- create official match result without official event
- silently ignore unknown event types

| DIRECTION_SWITCHED | yes | no | no | no | no | no | no | yes | no |
| MATCH_FINISHED | yes | no | no | no | yes | yes | yes | yes | maybe |
| MATCH_MARKED_OFFICIAL | yes | no | no | no | no | no | yes | yes | yes |
| CORRECTION_APPLIED | yes | yes | yes | yes | yes | yes | yes | yes | maybe |

## 8.2 Unknown Event Policy

If a projection builder sees an unknown event type:

```txt
1. Do not crash public scoreboard if safe fallback exists.
2. Mark projection warning.
3. Mark projection status as STALE if the event may affect state.
4. Require developer review.
5. Do not silently ignore it.
```

## 9. Rebuild Strategy

## 9.1 Full Rebuild

Full rebuild starts from empty projection state and applies all events in sequence.

```txt
1. Load all match_events ordered by seq_no
2. Start from initial projection state
3. Apply each event in sequence
4. Save projection state
5. Update checkpoint to latest seq_no
```

## 9.2 Rebuild From Snapshot

If snapshot exists:

```txt
1. Load latest snapshot for match
2. Load events after snapshot.last_seq_no
3. Apply events in order
4. Save projection state
5. Update checkpoint
```

## 9.3 When to Rebuild

Rebuild is required when:

- projection code changes
- projection version changes
- rule profile logic changes
- correction affects past state
- projection checkpoint is behind event stream
- unknown event type was detected
- admin manually triggers rebuild
- deployment migration changes read model shape

## 9.4 Rebuild Endpoint

Admin-only endpoint:

```http
POST /api/admin/projections/rebuild
```

Payload:

```json
{
  "scopeType": "MATCH",
  "scopeId": "match_123",
  "projectionNames": [
    "live_scoreboard_projection",
    "operator_score_projection",
    "foul_projection"
  ],
  "reason": "projection version upgrade"
}
```

Authorization:

```txt
permission: projection.rebuild
audit_required: true
reason_required: true
```

## 10. Polling-First Realtime Model

## 10.1 Why Polling-First

In Hostatom/Plesk shared hosting mode, long-running socket processes may be limited. Therefore, the official realtime model must work through HTTP polling.

## 10.2 Client Polling Flow

```txt
1. Client loads /api/matches/:id/live-state
2. Client stores lastEventSeq
3. Client polls /api/matches/:id/events?afterSeq=lastEventSeq
4. If new events exist, client requests updated projection or applies patch
5. If server says FULL_STATE_SYNC_REQUIRED, client reloads live-state
```

## 10.3 Recommended Polling Frequency

```txt
Operator screen: 300-500 ms during active match
Public scoreboard: 500-1000 ms during active match
Remote viewer: 1000-3000 ms
Admin dashboard: 2000-5000 ms
Standings page: 5000-15000 ms or manual refresh
```

## 10.4 Poll Response

```ts
type PollEventsResponse = {
  matchId: string;
  fromSeq: number;
  toSeq: number;
  events: MatchEvent[];
  projectionVersion: number;
  fullStateSyncRequired: boolean;
  serverNow: string;
};
```

## 10.5 Full State Sync Response

```ts
type LiveStateResponse = {
  matchId: string;
  state: LiveScoreboardProjection;
  lastEventSeq: number;
  serverNow: string;
};
```

## 10.6 Socket.IO Optional Layer

If Socket.IO is available, it is only a delivery layer.

Socket messages may notify clients:

```txt
MATCH_EVENTS_AVAILABLE
LIVE_STATE_UPDATED
FULL_STATE_SYNC_REQUIRED
```

Clients must still support polling and full sync.

## 11. Clock and Shot Clock Projection Policy

## 11.1 Deadline-Based Clock

The official projection stores:

```txt
remainingMs
startedAtServerTime
status
```

The browser displays ticking time locally based on `serverNow`.

## 11.2 No Backend Timer Loop

Do not implement:

```ts
setInterval(() => updateClockInDb(), 1000)
```

Instead:

```txt
GAME_CLOCK_STARTED stores remainingMs and startedAt
GAME_CLOCK_STOPPED computes remainingMs server-side
```

## 11.3 Expiration Handling

Because shared hosting may not run background jobs reliably, expiration can be detected by:

- operator client polling
- server calculation on next command
- display-side local countdown
- explicit command such as `GAME_CLOCK_EXPIRED`

[SYSTEM RECOMMENDATION] The official expiration event should be appended when the operator confirms or when a command handler detects the clock has reached zero.

- event interpretation changes

Projection migrations may rebuild read models from event store.

## 18. Acceptance Criteria

Projection model is accepted when:

- UI reads projection endpoints, not raw event streams by default.
- Every projection includes `lastEventSeq`.
- Every projection can be rebuilt from `match_events`.
- Projection updates do not mutate historical events.
- Projection rebuild is admin-only and audited.
- Hostatom mode works without background workers.
- Polling-first realtime works without Socket.IO.
- Public projection hides private audit data.
- Operator projection supports correction workflow.
- Replay projection preserves original and correction events.
- Tournament standings projection can be marked stale after match correction.
- Tests cover projection build, rebuild, correction, polling, and authorization.
